# 个人博客项目需求说明

版本：v0.1  
目标：6 小时内完成一个可展示的个人博客 MVP，后续逐步扩展为支持登录、评论、搜索、AI 自动化、小游戏和云端部署的完整博客系统。

## 1. 项目定位

本项目是一个综合型个人博客，既要能展示个人主页和学习笔记，也要能作为后续技术项目展示、文章发布、AI 自动化内容、小游戏入口和用户互动平台。

站点主人信息：

- 姓名：付江樊
- 学校：浙江大学
- 兴趣：长跑、唱歌、游戏
- 其他信息：暂时使用占位内容，后续补充

## 2. 技术路线

主线技术栈：

- 前端：React
- 后端：FastAPI
- 数据库：PostgreSQL
- 自动化：GitHub Actions
- 部署：云服务器
- 架构：前后端分离

推荐阶段性落地方式：

1. MVP 阶段先完成 React 页面和 FastAPI 接口骨架，使用本地模拟数据。
2. 第二阶段接入 PostgreSQL，把文章、评论、点赞、收藏、点踩等数据持久化。
3. 第三阶段接入邮箱登录和 GitHub OAuth 登录。
4. 第四阶段配置 Docker、GitHub Actions、云服务器部署和域名访问。
5. 第五阶段接入 AI 自动化内容、文章总结、小游戏和更多个性化模块。

## 3. 6 小时 MVP 验收目标

MVP 需要先做出一个可以演示的版本，重点是让项目“看得见、点得动、结构清楚、后续能扩展”。

MVP 必须包含：

- 个人主页：首页展示姓名、学校、兴趣和个人介绍。
- 学习笔记/文章：展示文章列表和文章详情摘要。
- 搜索：支持按文章标题、内容、标签进行前端搜索。
- 用户互动雏形：点赞、收藏、点踩按钮可以在页面上产生状态变化。
- 评论区雏形：可以展示评论入口和基础评论数据结构。
- AI 内容入口：展示每日技术新闻和文章 AI 总结模块的位置。
- 小游戏入口：预留并指向 `card-war-made-by-class-3` 决斗小游戏集成区域。
- 后端接口骨架：FastAPI 提供个人信息、文章列表、搜索、AI 内容等接口。
- 文档：更新版本日志和代码分析文档，方便后续验收答辩。

MVP 暂不强制完成：

- 真实注册登录
- GitHub OAuth
- PostgreSQL 持久化
- 云服务器正式部署
- GitHub Actions 自动部署
- 小游戏仓库真实嵌入
- AI 新闻和总结的真实模型调用

这些功能进入后续阶段完成。

## 5. 页面结构

建议前端页面分为以下模块：

- 首页：个人信息、最近文章、项目状态、兴趣展示
- 文章：文章列表、标签、搜索、文章详情入口
- AI：每日技术新闻、文章 AI 总结
- 游戏：决斗小游戏入口和排行榜预留
- 互动：评论、点赞、收藏、点踩
- 登录：邮箱登录和 GitHub 登录入口
- 管理：文章发布和内容管理入口，后续只对管理员开放

## 6. 数据模型初稿

后续数据库建议包含：

- `users`：用户表，存储邮箱、GitHub ID、昵称、头像、角色
- `articles`：文章表，存储标题、摘要、正文、标签、发布时间、作者
- `comments`：评论表，存储文章 ID、用户 ID、评论内容、发布时间
- `reactions`：互动表，存储点赞、收藏、点踩行为
- `tags`：标签表
- `article_tags`：文章和标签的关联表
- `ai_digest`：AI 总结和每日技术新闻缓存表
- `game_scores`：小游戏排行榜表

MVP 阶段先使用前端静态数据和后端内存数据，确认页面和接口结构后再接入 PostgreSQL。

## 7. 接口规划

FastAPI 初始接口：

- `GET /api/health`：健康检查
- `GET /api/profile`：获取个人信息
- `GET /api/articles`：获取文章列表，支持 `q` 搜索参数
- `GET /api/articles/{article_id}`：获取文章详情
- `POST /api/articles/{article_id}/comments`：提交评论
- `POST /api/articles/{article_id}/reaction`：提交点赞、收藏、点踩
- `GET /api/ai/news`：获取每日技术新闻占位数据
- `GET /api/game/card-war`：获取小游戏集成信息

后续登录接口：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/auth/github/login`
- `GET /api/auth/github/callback`
- `POST /api/auth/logout`

## 8. GitHub 和账号需求

第一步本地搭建不需要 GitHub 账号连接。

后续需要你提供或完成：

- GitHub 用户名
- 用于托管本项目的 GitHub 仓库地址
- 是否允许我帮你初始化本地 Git 仓库并绑定远程仓库
- GitHub OAuth App 的 Client ID 和 Client Secret，或者你授权我指导你创建
- GitHub Actions 部署用的服务器 SSH 私钥和服务器地址
- 目标云服务器信息：公网 IP、系统版本、SSH 用户名
- 域名信息：主域名、是否备案、DNS 服务商

敏感信息不要直接写入代码文件，后续统一放在 `.env` 或 GitHub Secrets 中。

## 9. 版本更新规范

每次版本更新后都需要：

1. 推送 GitHub。
2. 更新 `CHANGELOG.md`，说明该版本做了什么改动。
3. 更新 `CODE_ANALYSIS.md`，说明通过哪些代码实现了哪些功能。
4. 如果功能可运行，记录启动方式和验收方式。

当前本地还没有连接 GitHub 远程仓库，所以第一版先完成本地文件和文档。等你提供仓库地址后再进行远程推送。
