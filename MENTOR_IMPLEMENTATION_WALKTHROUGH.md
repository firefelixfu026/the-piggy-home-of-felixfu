# Mentor 演示实现说明

这份文档用于向 mentor 解释：个人博客每一个演示步骤背后的功能是如何实现的。它和 `MENTOR_REVIEW.md` 的区别是：

- `MENTOR_REVIEW.md` 偏向“怎么演示”。
- 本文档偏向“功能是怎么做出来的、代码和系统结构如何支撑”。

项目地址：

- 线上站点：`https://www.felixfu.xyz`
- GitHub 仓库：`https://github.com/firefelixfu026/the-piggy-home-of-felixfu`
- 主要技术栈：React + Vite、FastAPI、PostgreSQL、Docker Compose、Nginx、GitHub Actions

## 1. 首页和整体入口

### 演示内容

打开首页，介绍个人信息、项目定位、核心能力入口和当前项目阶段。

### 实现方式

首页由前端 React 单页应用负责渲染，核心文件在：

- `frontend/src/App.jsx`
- `frontend/src/styles.css`
- `frontend/src/data.js`

前端通过组件状态控制当前页面视图，例如首页、文章页、AI 页、游戏页、登录页、账号中心和管理后台。左侧导航不是多个独立 HTML 页面，而是 React 根据当前状态切换不同的工作区组件。

首页展示的数据分为两类：

1. 静态展示数据：如个人名称、阶段介绍、能力卡片等，主要来自 `frontend/src/data.js` 或 React 组件中的配置。
2. 动态数据：如文章、统计、账号状态等，会通过 `/api/...` 接口向后端请求。

### 设计重点

- 首页不是纯静态页面，而是整个系统的入口。
- 不同登录状态看到的导航不同。
- 游客不会看到管理后台入口，管理员登录后才会出现管理入口。

## 2. 前后端分离结构

### 演示内容

说明这个博客不是简单网页，而是一个完整的前后端分离项目。

### 实现方式

项目结构大致如下：

```text
frontend/              React + Vite 前端
backend/               FastAPI 后端
docker-compose.yml     本地和服务器统一启动配置
.github/workflows/     GitHub Actions 自动测试和部署
docs/                  归档和辅助文档
*.md                   部署、数据库、运维和验收文档
```

前端负责：

- 页面展示
- 表单交互
- Markdown/LaTeX 渲染
- 登录状态保存
- 调用后端接口

后端负责：

- 提供 REST API
- 用户认证
- 权限判断
- 数据库读写
- 图片上传
- GitHub OAuth 回调
- AI 接口调用和本地回退

数据库负责：

- 保存文章
- 保存用户
- 保存评论
- 保存点赞、收藏、点踩、问号互动
- 保存上传图片记录
- 保存 AI 生成历史

## 3. 文章列表、搜索、标签和分类

### 演示内容

进入文章页，展示文章列表、搜索、分类筛选、标签筛选、月份归档和热门文章。

### 实现方式

文章数据存储在 PostgreSQL 中，由后端模型 `Article` 管理：

- `backend/app/models.py`

后端文章接口主要在：

- `backend/app/main.py`

前端文章展示逻辑主要在：

- `frontend/src/App.jsx`

文章列表接口会从数据库读取文章，再返回 JSON 给前端。前端收到文章列表后，在浏览器中根据搜索词、标签、分类和归档月份进行筛选展示。

文章字段包括：

- 标题
- 摘要
- 正文
- 标签
- 日期
- 阅读时长
- 分类
- 是否置顶
- 状态：草稿或已发布
- 阅读量

### 设计重点

- 文章不是写死在前端，而是来自数据库。
- 分类和标签让内容可以长期沉淀。
- 搜索支持标题、正文和标签关键词。
- 置顶文章会在列表中优先展示。

## 4. 文章详情、Markdown 和 LaTeX

### 演示内容

打开文章详情，展示正文内容、Markdown 格式、LaTeX 公式和文章图片。

### 实现方式

文章正文以 Markdown 文本形式存储在数据库中。前端渲染时使用 Markdown 渲染组件，将 Markdown 转换成 HTML。

LaTeX 公式支持通过 KaTeX 相关依赖实现。构建输出中可以看到 KaTeX 字体资源，说明公式渲染依赖已经进入前端构建产物。

前端会对文章正文中的内容进行处理：

- Markdown 标题、列表、引用、代码块
- 行内公式和块级公式
- 图片链接
- 普通段落和换行

### 设计重点

- 文章正文适合写学习笔记、技术总结和课程内容。
- LaTeX 支持让数学公式、算法公式和课程笔记可以正常展示。
- 图片和公式都在文章详情中统一渲染。

## 5. 文章图片上传和正文图片

### 演示内容

在管理后台上传图片，并把图片插入文章正文中。

### 实现方式

图片上传由后端 FastAPI 接收文件，保存到服务器的上传目录。Docker Compose 中将上传目录挂载为持久化 volume：

- `uploads_data:/app/uploads`

后端通过静态文件服务暴露上传图片，让前端可以通过 URL 访问。

相关实现位置：

- `backend/app/main.py`
- `docker-compose.yml`
- `frontend/src/App.jsx`

前端上传图片后，会获得图片 URL。管理员可以将该 URL 插入文章正文，Markdown 渲染器再把它展示为文章图片。

### 设计重点

- 图片不是 base64 写入正文，而是作为独立文件保存。
- 上传目录通过 Docker volume 持久化，容器重建后图片不丢。
- 管理后台可以查看和管理已经上传的图片。

## 6. 登录系统和用户身份

### 演示内容

展示邮箱登录、GitHub OAuth 登录、登录后的用户状态和账号中心。

### 实现方式

登录认证由后端负责，核心文件包括：

- `backend/app/auth.py`
- `backend/app/github_oauth.py`
- `backend/app/main.py`
- `backend/app/models.py`

系统支持两类登录：

1. 邮箱和密码登录
2. GitHub OAuth 登录

邮箱登录流程：

1. 用户输入邮箱和密码。
2. 后端校验密码哈希。
3. 校验成功后生成 token。
4. 前端将 token 保存到浏览器本地。
5. 后续请求通过 Authorization header 携带 token。

GitHub OAuth 流程：

1. 前端跳转到 GitHub 授权页面。
2. GitHub 登录成功后回调后端接口。
3. 后端用 code 换取 GitHub 用户信息。
4. 如果是首次登录，则创建或更新本地用户。
5. 后端生成本站 token，再跳回前端。

### 设计重点

- 登录状态由后端 token 保护。
- GitHub 登录名可以作为评论和账号显示名称。
- 管理员和普通用户通过 `role` 字段区分。

## 7. 管理员权限和后台入口隐藏

### 演示内容

游客看不到管理入口，管理员登录后才能看到管理入口并进入后台。

### 实现方式

权限控制分两层：

1. 前端显示控制
2. 后端接口保护

前端根据当前用户 `role` 判断是否显示管理入口。普通用户和游客即使知道页面入口，也不能真正调用后台接口。

后端通过 `require_admin` 依赖保护管理员接口，例如：

- 发布文章
- 编辑文章
- 删除文章
- 评论管理
- 图片管理
- 站点统计
- AI 设置测试

相关实现：

- `backend/app/auth.py`
- `backend/app/main.py`
- `frontend/src/App.jsx`

### 设计重点

- 隐藏入口只是用户体验层。
- 真正的安全来自后端接口鉴权。
- 初始化管理员需要 `ADMIN_SETUP_TOKEN`，避免任何人都能创建管理员。

## 8. 文章发布、编辑、草稿和置顶

### 演示内容

在管理后台发布文章、编辑文章、保存草稿、设置分类、标签和置顶。

### 实现方式

文章表单由前端管理后台组件实现。管理员填写表单后，前端向后端发送请求。

后端根据请求创建或更新 `Article` 数据表记录。文章状态字段用于区分：

- 已发布
- 草稿

置顶字段用于控制文章列表排序。后端读取文章时会优先返回置顶文章，再按创建时间排序。

### 设计重点

- 后台不是临时表单，而是完整 CMS 工作流。
- 草稿让管理员可以保存未完成内容。
- 分类和标签提升内容管理能力。
- 置顶适合展示重要文章或项目说明。

## 9. 评论系统、回复、分页和审核

### 演示内容

登录后发表评论，评论支持分页；管理员可以在后台管理评论。

### 实现方式

评论数据由 `Comment` 模型保存，包含：

- 评论内容
- 作者名称
- 用户 ID
- 文章 ID
- 回复目标 ID
- 审核状态
- 创建时间

相关实现：

- `backend/app/models.py`
- `backend/app/main.py`
- `frontend/src/App.jsx`

用户发表评论时，后端会检查：

1. 是否登录。
2. 评论内容是否为空。
3. 评论内容是否超过长度限制。
4. 是否触发评论频率限制。
5. 如果是回复，回复目标是否存在。

前端展示时会进行分页，避免评论太多导致页面过长。管理后台的评论列表也做了分页和限宽，防止超长文本撑破布局。

### 设计重点

- 游客不能评论。
- 评论有字数限制。
- 评论有频率限制。
- 评论可以回复。
- 管理员可以集中管理评论。

## 10. 点赞、收藏、点踩和问号互动

### 演示内容

登录后可以点赞、收藏、点踩和点击问号互动；刷新页面不会重复计数。

### 实现方式

互动不是简单前端计数，而是账号级持久化记录。

后端模型中包含：

- `ReactionCounter`：保存每篇文章每种互动的总数。
- `UserReaction`：保存某个用户对某篇文章做过哪种互动。

用户点击互动按钮时：

1. 前端发送文章 ID 和互动类型。
2. 后端检查用户是否登录。
3. 后端检查该用户是否已经对这篇文章做过同类互动。
4. 如果没有做过，则写入 `UserReaction`，并更新 `ReactionCounter`。
5. 如果已经做过，则不会重复增加计数。

### 设计重点

- 每个账号对同一篇文章同一种互动只记录一次。
- 刷新页面不会重复计数。
- 这体现了后端数据一致性，而不是前端假计数。

## 14. AI 工作台

### 演示内容

进入 AI 页面，输入主题，生成选题、摘要或标题建议。

### 实现方式

AI 工作台前端表单收集：

- 生成模式
- 主题
- 正文参考内容
- 标签
- 语气

然后调用后端 `/api/ai/workbench` 接口。后端会判断是否配置真实模型：

1. 如果配置了真实模型，则调用 AI Provider。
2. 如果没有配置，或者真实模型调用失败，则使用本地回退生成逻辑。

本地回退逻辑由后端根据输入生成稳定的候选内容，保证演示流程永远可用。

### 设计重点

- AI 功能有真实模型路径。
- AI 功能也有本地回退路径。
- API 异常不会导致整个网站不可用。

## 15. AI 写作辅助和历史记录

### 演示内容

在管理后台文章编辑页使用 AI 润色、续写或生成大纲，并查看 AI 历史。

### 实现方式

管理员在后台文章表单中点击 AI 操作按钮，前端会将当前标题、摘要、正文和任务类型发送到后端。

后端根据任务类型构造 prompt：

- 润色
- 续写
- 大纲

生成结果返回后，前端可以把结果追加到正文中。后端还会将 AI 生成记录保存到 `AiGeneration` 数据表，刷新后仍能看到历史记录。

### 设计重点

- AI 不只是单次弹窗，而是进入文章写作流程。
- AI 结果有来源标记：真实模型、本地回退或失败回退。
- AI 历史持久化，便于追踪使用情况。

## 16. AI Provider 和中转站接入

### 演示内容

在后台 AI 设置和测试面板测试真实模型连通性。

### 实现方式

系统支持两种 AI 接口模式：

1. `openai`：OpenAI 兼容接口。
2. `codex`：Codex 中转接口。

相关配置项：

```text
AI_PROVIDER_NAME=
AI_API_STYLE=
AI_BASE_URL=
AI_MODEL=
AI_API_KEY=
AI_REQUEST_TIMEOUT=
```

如果中转站提供的是 OpenAI SDK 兼容地址，例如：

```text
https://api.aicodemirror.com/api/codex/backend-api/codex/v1
```

则后台选择：

```text
接口类型：OpenAI 兼容
Base URL：https://api.aicodemirror.com/api/codex/backend-api/codex/v1
```

后端会自动拼接：

```text
/chat/completions
```

最终请求：

```text
https://api.aicodemirror.com/api/codex/backend-api/codex/v1/chat/completions
```

如果中转站只提供完整 Codex 接口，则选择 `Codex 中转`，后端会直接请求填写的完整地址，不自动拼接 `/chat/completions`。

### 安全设计

- 后台测试框中的 API Key 只用于一次性连通性测试。
- API Key 不会从接口返回给前端。
- 长期配置应写入服务器环境变量，不写入 GitHub 仓库。
- 如果接口返回 HTML 页面，后端会提示 Base URL 填错，避免把网页源码当成模型输出。

## 18. 数据库持久化

### 演示内容

说明刷新页面、重启服务后文章、评论、用户和图片不会丢失。

### 实现方式

数据库使用 PostgreSQL。后端通过 SQLAlchemy 定义模型并访问数据库。

主要模型包括：

- `Article`
- `Tag`
- `Comment`
- `User`
- `ReactionCounter`
- `UserReaction`
- `UploadedImage`
- `AiGeneration`

Docker Compose 中 PostgreSQL 使用 volume 保存数据：

```yaml
postgres_data:/var/lib/postgresql/data
```

### 设计重点

- 数据不是存在浏览器里。
- 容器重启后数据库数据仍然保留。
- 后端启动时会自动做基础表结构检查和补齐。

## 19. Docker Compose 一键启动

### 演示内容

说明本地和服务器都可以通过 Docker Compose 统一启动。

### 实现方式

`docker-compose.yml` 定义了三个服务：

1. `postgres`：数据库
2. `backend`：FastAPI 后端
3. `frontend`：React 构建后的静态站点

服务之间通过 Docker 网络互相访问。后端连接 PostgreSQL，前端通过 Nginx 或容器网络访问后端接口。

### 设计重点

- 降低部署复杂度。
- 本地环境和服务器环境更一致。
- 数据库、后端、前端可以一条命令启动。

## 20. Nginx 和 HTTPS

### 演示内容

说明线上网站通过域名和 HTTPS 访问。

### 实现方式

服务器上使用 Nginx 作为反向代理：

- 用户访问 `https://www.felixfu.xyz`
- Nginx 接收请求
- 静态页面请求转发到前端容器
- `/api/...` 请求转发到后端容器
- 上传图片路径转发到后端静态文件服务

HTTPS 证书由 Certbot / Let's Encrypt 配置和续期。

### 设计重点

- 前端、后端和上传资源统一挂到同一个域名下。
- 用户不需要直接访问容器端口。
- HTTPS 提升正式部署可用性。

## 21. GitHub Actions 自动测试和部署

### 演示内容

打开 GitHub Actions，展示 CI 和 Deploy 成功记录。

### 实现方式

工作流文件位于：

- `.github/workflows/ci.yml`
- `.github/workflows/deploy.yml`

CI 主要检查：

- 后端 Python 编译
- FastAPI 应用导入
- 前端构建
- Docker Compose 配置

Deploy 在 CI 成功后触发，通过 SSH 连接服务器执行部署脚本：

1. 服务器拉取最新代码。
2. 重新构建或重启 Docker 服务。
3. 检查容器状态。
4. 更新线上站点。

### 设计重点

- 每次 push 后自动检查。
- 检查通过后自动部署。
- 减少手动登录服务器操作。

## 22. 备份和运维

### 演示内容

说明项目不是只跑起来，还考虑了后期维护。

### 实现方式

- 有备份意识。

- 适合长期维护，而不是一次性演示项目。

## 23. 国内访问和 ICP 限制

### 演示内容

说明部分国内网络或微信内置浏览器可能访问异常。

### 实现方式和限制

网站部署在阿里云中国香港服务器，可以正常公网访问。但是国内平台或微信内置浏览器可能因为 ICP 备案状态拦截未备案网站。

这是外部合规限制，不是代码问题。

### 设计重点

- 海外或香港节点不一定要求 ICP 才能技术访问。
- 但国内平台和微信可能按自己的策略拦截。
- 如果要让国内手机和微信稳定访问，需要完成 ICP 备案或使用符合备案要求的服务器和域名配置。

## 24. 安全设计总结

系统目前做了以下安全加固：

- 管理入口只对管理员显示。
- 后端管理员接口用 `require_admin` 保护。
- 初始化管理员需要 `ADMIN_SETUP_TOKEN`。
- 游客不能评论和互动。
- 点赞、收藏、点踩、问号按账号去重。
- 评论有长度限制和频率限制。
- API Key 不写入仓库。
- GitHub OAuth Secret 使用环境变量。
- AI 测试框不会把密钥返回给前端。
- 上传图片限制 MIME 类型和文件后缀。
